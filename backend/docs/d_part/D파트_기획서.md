# D파트 기획서: 전세사기 및 특수상황 대응

## 1. 개요

### 1.1 배경
한국 법률 정보를 비전문가도 이해할 수 있도록 제공하는 멀티에이전트 RAG 시스템 프로젝트의 일부로, D파트는 **주택임대차 도메인 중 전세사기 및 특수상황**을 담당한다.

### 1.2 D파트가 어려운 이유
- 구현 복잡도(C파트) 문제가 아니라 **도메인 지식, 데이터 희소성(신법 판례 부족), 사기 의도에 대한 LLM 추론**이 핵심 난이도
- D파트와 C파트는 표면적으로 유사해 보이지만 법적 프레임워크가 다름. **팀 결정에 따라 MVP 단계에서는 경계 문서화를 별도로 진행하지 않고, 각 파트 독자 구현 후 파트 간 에이전트 통신을 통해 경계를 조율하는 방식을 Post-MVP에서 검토 예정 (섹션 6 참고)**

### 1.3 공통 원칙 (프로젝트 전체 적용)
- 모든 응답은 **원문 → 해설 → 상황 적용** 3단 구조
- 판단은 **높음 / 있음 / 추가확인** 3단계 가능성 언어로 표현 (단정적 판단 금지)
- 면책 조항 자동 삽입
- 시각 자료/다이어그램 없이 텍스트로만 응답

## 2. 담당 범위

### 2.1 계약 단계 재정의 (팀 회의 확정)
| 단계 | 정의 |
|---|---|
| 전 | 계약서를 작성하기 전 + 갓 작성 완료한 상태 |
| 중 | 실제 거주하고 있는 상태 |
| 후 | 계약이 끝나서 나가야 하는 상태 |

D파트는 각 단계에서 **전세사기 관련 위험/리스크 관점**으로 범위를 좁혀 담당한다. (일반 임대차 절차 안내는 D파트 범위 아님)

#### 전 (계약서 작성 전~직후) — 확정
6가지 위험 유형을 다루되, **일반적인 부동산 상식 설명이 아니라 실제 전세사기 판례/사례집 기반 사례를 근거로 "이런 식으로 사기가 발생했다"는 패턴을 먼저 제시한 뒤, 사용자 상황에 어떤 위험이 있는지 짚어주는 방식**으로 응답한다. (1.3 원문→해설→상황적용 구조에서 "해설"이 사례 기반 패턴 설명에 해당)

1. 등기부등본 위험 신호 해석 (근저당, 가압류, 소유권 이전 이력)
2. 전세가율/HUG 보증보험 가입 가능 여부 미확인
3. 다가구주택 선순위 보증금 미확인
4. 계약서 특약사항 위험 조항 해석
5. 임대인 실제 소유자 여부(신탁사기)
6. 공인중개사 허위/누락 고지

#### 중 (실거주 중) — 확정
"중" 단계의 역할은 **이미 발생한 위험 신호에 반응하는 것(2.1.1 트리거 로직)과 구분**된다 — 아직 위험 신호가 표면화되지 않은 상태에서, 거주 중 능동적으로 확인해야 할 체크포인트를 예방적으로 안내하는 것이 핵심이다.

"전" 단계와 동일하게, **일반적인 상식 설명이 아니라 실제 전세사기 판례/사례집 기반 사례를 근거로 "이런 식으로 사기가 발생했다"는 패턴을 먼저 제시한 뒤, 사용자 상황에 어떤 위험이 있는지 짚어주는 방식**으로 응답한다. (1.3 원문→해설→상황적용 구조에서 "해설"이 사례 기반 패턴 설명에 해당)

1. 소유권 변동 모니터링 — 거주 중 임대인이 바뀌거나 매매/경매로 소유권이 넘어가는 경우 확인 방법과 위험 신호
2. 근저당 추가 설정 모니터링 — 계약 당시엔 없던 근저당이 거주 중 새로 설정되는 경우
3. 임대인 세금 체납 확인 — 국세/지방세 체납으로 인한 압류 위험
4. 갱신 시점 위험 — 계약 갱신 과정에서 발생하는 갱신료 요구, 갱신 거절을 빙자한 사기 패턴
5. 다가구주택 타 세입자 피해 소식 — 같은 건물 내 다른 세입자 피해를 조기 경보 신호로 해석

> ※ 위 체크포인트 확인 중 실제 위험 신호가 감지되면 2.1.1 트리거 로직으로 전환

#### 후 (계약 종료/퇴거) — 확정
"후" 단계의 핵심 로직은 **2.1.1 미인지형/인지형 판별 로직**이며, 보증금 미반환·임대인 사망/파산·형사고소 관련 대응은 2.1.1과 특수상황(2.2 임대인 사망/파산, 형사고소/무고죄)에서 다룬다.

"후" 단계 고유 항목은 아래 2가지로, 마찬가지로 **실제 전세사기 판례/사례집 기반 사례를 근거로 패턴을 먼저 제시한 뒤 사용자 상황의 위험을 짚어주는 방식**으로 응답한다.

1. 대항력/우선변제권 상실 위험 — 임차권등기명령 없이 퇴거해 대항력을 잃는 흔한 실수 패턴
2. 이중계약/배당 순위 다툼 — 같은 집을 여러 세입자와 계약한 경우 배당 순위에서 밀리는 패턴

> ※ 위 항목 확인 중 미인지/인지 판별이 필요한 상황이 감지되면 2.1.1 로직으로 전환

### 2.1.1 위험 신호 감지 트리거 로직 (단계 횡단, cross-cutting)
미인지형 판별(피해자법 요건 판단)은 "후" 단계 전용이 아니라, **계약 단계와 무관하게 위험 신호가 감지되면 즉시 발동**하는 구조로 설계한다.

**트리거 조건 (발화 내 위험 신호)**
- 경매/공매 개시 통지를 받았다는 언급
- 보증금을 돌려받지 못하고 있다는 언급
- 임대인과 연락이 안 된다는 언급
- 임대인 사망/파산 소식을 들었다는 언급
- 소유권 변동(경매 낙찰, 명의 이전)을 알게 됐다는 언급
- "같은 집주인에게 다른 세입자도 피해를 봤다"는 언급

**흐름**
1. 분류기가 계약 단계(전/중/후) 1차 판별 (3.2)
2. 단계 판별과 별개로 **위험 신호 감지 노드**가 발화를 스캔
3. 위험 신호 감지 시 → 단계 무관하게 미인지형 판별 로직(결합형 슬롯 채우기)으로 라우팅
4. 신호 없으면 → 해당 단계의 일반 시나리오 로직으로 진행

**미인지형/인지형 구분 및 판별 로직**

| 유형 | 설명 |
|---|---|
| 미인지형 | 본인이 전세사기 피해자인지 모르는 상태. 상황 설명을 바탕으로 피해자법 요건 해당 가능성을 판별 |
| 인지형 | 이미 피해자임을 인지한 상태. 요건 충족 확인 + 지원 절차 안내(우선매수권, 경공매 유예, 조세채권 안분 등) |

**미인지형 판별 로직 (결합형)**
1. 사용자가 상황을 자유 서술
2. LLM이 서술 내용을 전세사기피해자법 요건 슬롯(예: 임대차 종료 여부, 보증금 반환 여부, 경공매 개시 여부, 임대인 상습성 등)에 매핑 → 슬롯별 충족/불충족/불명확 판정
3. 불명확한 슬롯만 추가 질문으로 보완
4. 모든 슬롯 충족 확인 시 판단 결과 제공(높음/있음/추가확인), 반복 질문에도 채워지지 않으면 Fallback(전문가 상담 안내)

> 요건 슬롯 목록은 전세사기피해자법 조문 확인 후 확정 예정 (섹션 5, 7 참고)

### 2.2 특수상황 7가지
1. 임대인 사망/파산
2. 신탁사기
3. 다가구주택
4. 유주택자
5. 형사고소/무고죄
6. 공인중개사 허위고지
7. (7번째 항목 확인 필요)

### 2.3 Fallback 정책
범위를 벗어나는 케이스는 전문가 상담으로 안내

### 2.4 D파트 ↔ C파트 병행 처리 (논의 보류)
- 전세사기 피해자가 경매 절차에 얽히는 경우, D파트와 C파트가 병행 실행되는 시나리오 존재
- 근거: 전세사기 피해자가 경매를 만나는 경우는 전세사기피해자법에 따라 D파트 소관
- **팀 결정(2026-07)**: 각 파트는 다른 파트와의 중복/경계 문제를 우선 고려하지 않고 독자적으로 구현 진행. 본 항목은 MVP 단계에서 급하게 결정하지 않고, D파트 관점에서 우선 구현 후 통합 단계에서 재논의

## 3. 아키텍처

### 3.1 LangGraph 서브그래프 구조
**처리 로직 노드** (라우팅/분류 역할)
- 분류기
- 전 / 중 / 후 (2.1 정의 기준)
- 위험 신호 감지 노드 (단계 횡단, 2.1.1)
- 특수상황

**실제 에이전트** (작업 수행)
- OCR 에이전트
- RAG 에이전트
- 응답생성 에이전트

### 3.2 계약 단계 판별 및 위험 신호 감지 흐름
1. 분류기가 사용자 입력을 기반으로 계약 단계(전/중/후) 자동 판별
2. 판별 결과를 사용자에게 확인받는 단계 거침
3. 위험 신호 감지 노드가 발화 내 위험 신호(2.1.1 트리거 조건) 스캔
4. 위험 신호 감지 시 → 단계 무관 미인지형 판별 로직으로 라우팅
5. 신호 없으면 → 확인된 단계의 일반 시나리오 로직으로 진행

### 3.3 페르소나별 톤 적용
- 공감 우선(empathy-first) 톤은 **페르소나 C, D에만 적용** (전체 D파트 사용자 대상 아님)

## 4. 데이터 소스 및 RAG 설계

### 4.1 소스 매핑
| 소스 | 성격 | 비고 |
|---|---|---|
| 전세사기피해자법 | 법령 조문 | law.go.kr API |
| 공인중개사법 | 법령 조문 | law.go.kr API |
| HUG 관련 규정 | 스크래핑 | 자체 사이트 |
| HUG 전세피해지원 사례집 PDF | Q&A 22건 (14p~) | 구조화된 케이스 |
| 판례 | 익명화 공개 자료 | law.go.kr API |

### 4.2 핵심 데이터 이슈
- **법령해석례 부족**: 임대차 관련 질의에 대해 law.go.kr API 상 법령해석례가 거의 없음 → 판례가 주 근거, 부처 유권해석 및 현행 조문으로 보완
- **판결문 vs 판례 구분**: 판결문(비공개) / 판례(익명화, API 공개) — RAG 소스는 판례만 사용 가능, 데이터 가용성에 직접 영향

### 4.3 소스별 파싱/청킹 전략 (초안)
| 소스 | 청킹 단위 | 핵심 메타데이터 |
|---|---|---|
| 법령 조문 | 조 단위 (항 많으면 항 단위 분리) | 법령명, 조번호, 시행일자 |
| 판례 | 판결요지 우선, 필요시 전문 사실관계/판단 분리 | 사건번호, 선고일자, 참조조문 |
| HUG 규정 | 섹션 기준, 없으면 슬라이딩 윈도우 | 출처 URL, 수집일자, 카테고리 |
| 사례집 PDF | 케이스(Q+A) 단위 = 1청크 | 케이스번호, 주제태그 |

> 조문-판례-사례집을 참조조문/주제태그로 링크해 응답생성 시 "원문→해설→상황적용" 구조 지원

## 6. Post-MVP

- **파트 간 에이전트 통신 (D-C파트 경계 조율 포함)** — ⚠️ MVP 범위 아님, 지금 단계에서 구현하지 말 것. MVP는 파트별 독립 DB + 독자 처리로 완료 후 검토
- MCP 통합 (MVP 이후)
- 첫 MCP 유스케이스: **보증금 반환 최고서 자동 작성** (대화 State 컨텍스트 활용)

## 8. 인증 및 대화 이력 정책

### 8.1 회원가입
- 소셜 로그인 없이 **자체(로컬) 회원가입만 지원**
- 수집 항목은 **닉네임, 아이디, 비밀번호** 세 가지로 최소화 (그 외 개인정보 미수집)
- 중복 가입 방지는 **아이디 유니크 제약**으로 처리 (닉네임 중복 허용 여부는 미정)

### 8.2 대화 이력 저장 정책
| 사용자 유형 | 대화 이력 |
|---|---|
| 비로그인 | 저장하지 않음 |
| 로그인 | 사용자별 DB에 저장 |

### 8.3 화면 구성 (좌측 사이드바)
- 생성형 AI 서비스(GPT, Gemini, Claude 등)처럼 좌측에 최근 대화 목록을 요약 형태로 표시
- 요약을 클릭하면 해당 대화를 로드
- 사이드바 상단부터 순서대로: ① 회원정보(닉네임) ② 로그아웃 버튼 ③ 대화 요약 목록

### 8.4 대화 요약 로직
- 대화 요약(사이드바 표시용 제목) 생성에 **Ollama 기반 로컬 모델 도입 검토**
